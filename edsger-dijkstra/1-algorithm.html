I. The Shortest Path · 1956

In the spring of 1956, Dijkstra was sitting in a café in Amsterdam — the Café Americain, by some accounts — with his fiancée. He was thinking about a problem: how to find the shortest route between two cities in a road network. He had no paper. He thought for about twenty minutes, then he had it. He would not publish the result until 1959, in the journal Numerische Mathematik. The algorithm he described in that paper is now called Dijkstra's algorithm, and it runs inside every GPS navigation system, every internet routing protocol, every logistics optimizer on earth.

The elegance of the algorithm is that it is greedy in the mathematically precise sense: at each step, it commits to the nearest unvisited node, knowing that this choice will never be regretted. The proof of correctness is simple — if the edge weights are non-negative. Dijkstra thought of this later as one of his cleaner inventions: a solution that could be explained in five minutes, proved correct in ten, and implemented in an afternoon.

How It Works — Step by Step
  • Initialize. Assign distance 0 to the source node S. Assign distance ∞ to every other node. Mark all nodes as unvisited. Put all nodes in a priority queue ordered by distance.
  • Select. Extract the unvisited node u with the minimum current distance from the priority queue. (First iteration: this is the source.)
  • Relax neighbors. For each unvisited neighbor v of u: compute dist[u] + weight(u,v). If this is less than dist[v], update dist[v] and record u as v's predecessor.
  • Mark visited. Mark u as visited. A visited node is permanently settled — its shortest distance will not change. This is the greedy commitment.
  • Repeat. Return to step 2. Continue until the destination node is extracted, or all reachable nodes are visited.
  • Recover path. Follow predecessor pointers from destination back to source to reconstruct the shortest path. The distance at the destination node is the shortest-path distance.

For the graph shown in the hero above — six nodes S, A, B, C, D, E — the algorithm finds S→A→C→E with total weight 4+3+5=12. Every GPS in the world does a version of this, on graphs with millions of nodes, in under a second.

"What is the shortest way to travel from Rotterdam to Groningen, in general: from given city to given city. It is the algorithm for the shortest path, which I designed in about twenty minutes." — Edsger W. Dijkstra, 2001 interview
II. Go To Statement Considered Harmful · 1968

In 1968, Dijkstra sent a short letter to Communications of the ACM. It was published in March of that year under the title "Go To Statement Considered Harmful" — a title added by the editor, Niklaus Wirth. The letter argued, concisely and devastatingly, that the GOTO statement — which allowed a program to jump to any arbitrary location in the code — was incompatible with writing programs that humans could understand and verify.

The argument was not merely aesthetic. Dijkstra showed that a program's runtime behavior could be understood by a programmer only if the progress of the computation could be traced through the program's static text. GOTO breaks this: after a jump, the reader no longer knows where they are in the logic. The program text and the execution become decoupled. Understanding becomes impossible. Correctness proofs become impossible.

"The quality of programmers is a decreasing function of the density of go to statements in the programs they produce." — Edsger W. Dijkstra, "Go To Statement Considered Harmful," CACM, 1968

The backlash was immediate and furious. Many programmers had been using GOTO freely for a decade; Dijkstra was telling them they had been doing it wrong. Letters flooded CACM. Some programmers argued that GOTO was essential for efficiency, for handling error conditions, for real-world code. Dijkstra replied to some of them; he ignored others. He had said what he believed to be true. The fact that it was uncomfortable changed nothing.

Structured programming proposed that all program logic could be expressed with just three constructs: sequence (do A then B), selection (if-then-else), and iteration (while loops). Böhm and Jacopini had already proved this mathematically. Dijkstra was arguing for its adoption as engineering practice — and as moral practice. Programs written this way could be read, followed, and proved. Programs with GOTOs could not.

III. THE Multiprogramming System · 1968

In Eindhoven, Dijkstra and his students built what became known as the THE multiprogramming system — THE being an abbreviation of Technische Hogeschool Eindhoven, the Dutch name for the university. The THE system was one of the first operating systems designed with explicit layers of abstraction: each layer used only the services of the layer below, and exposed a clean interface to the layer above.

This hierarchical decomposition — which we now take entirely for granted in every operating system, every application framework, every network stack — was a radical departure from how systems had been built. Previously, systems were ad hoc accumulations of interacting parts, each touching every other. Dijkstra showed that this was not just inelegant but dangerous: in such systems, you could never be sure that a change in one place would not break something somewhere else.

The THE system demonstrated that an operating system could be designed so that each layer could be verified independently. This made correctness tractable. It is the intellectual ancestor of every layered software architecture built since — including TCP/IP, the UNIX kernel, and every web framework.

IV. Semaphores and Dining Philosophers · 1965

As computers began to run multiple processes simultaneously in the early 1960s, a new problem emerged: what happens when two processes need to access the same resource at the same time? Dijkstra invented the semaphore as a solution — a simple integer variable with two atomic operations: P (Dutch: proberen, to test) decrements the count; V (verhogen, to increment) increments it. If a process tries to P a semaphore at zero, it blocks until another process V's it.

With this simple mechanism, Dijkstra showed how to coordinate concurrent processes without race conditions. Semaphores are the foundation of every mutex, every monitor, every lock in every operating system and concurrent programming language written since.

To illustrate the subtleties of concurrency, he invented the Dining Philosophers problem: five philosophers sit at a round table. Each has a plate of spaghetti and a fork on either side. To eat, a philosopher needs both the fork on their left and the fork on their right. Philosophers alternate between thinking and eating. What rules should they follow to avoid deadlock — the state where every philosopher is holding one fork and waiting for the other, forever?

"The Dining Philosophers is one of the classic problems in concurrent programming. It is elegant precisely because it is simple enough to state in one paragraph but subtle enough to have occupied researchers for decades." — Introductory note in most operating systems textbooks, paraphrasing Dijkstra
V. Guarded Commands · 1975

Dijkstra introduced the guarded command language in 1975 as a formal notation for programs that was designed from the ground up to support correctness proofs. A guarded command has the form B → S: if condition B is true, execute statement S. Multiple guarded commands can be combined into a non-deterministic choice (if any guard is true, choose any of the true ones) or a loop (repeat until no guard is true).

The non-determinism was deliberate. By not specifying which branch to take when multiple guards are true, the language forces the programmer to prove that all choices lead to the same result — that the program is correct regardless of which path the runtime takes. This is a much stronger form of correctness than testing.

VI. Predicate Transformers · 1975

In his 1976 book A Discipline of Programming, Dijkstra formalized the concept of the weakest precondition. For any statement S and any desired postcondition Q (what should be true after S executes), the weakest precondition wp(S,Q) is the largest set of initial states from which execution of S is guaranteed to terminate with Q true.

This gave programmers a mechanical way to derive programs from specifications: start with what you want to be true at the end, apply wp transformations backwards through your statements, and you arrive at what must be true at the beginning. If the precondition is achievable, the program is correct by construction. You did not write a program and check it — you derived the program from the proof.

"A programming language is a tool that has a profound influence on our thinking habits." — Edsger W. Dijkstra, EWD 498
