The Algorithm Is Everywhere

Every time you ask Google Maps for directions, Dijkstra's algorithm runs. Every time your phone finds a route on Waze, Dijkstra's algorithm runs. Every time a logistics company optimizes a delivery route, Dijkstra's algorithm runs. Every time an internet router decides which path to forward a packet through the network — using OSPF, the Open Shortest Path First protocol — a variant of Dijkstra's algorithm runs.

The algorithm was published in 1959, in a nine-page paper in a German mathematics journal. It is now embedded in infrastructure that serves billions of people daily. The GPS in every car. The routing tables in every internet router. The logistics systems that move every package. Dijkstra invented it in twenty minutes at a café, without paper. He thought it was a rather nice result. He had no idea.

"Computer science is no more about computers than astronomy is about telescopes." — Edsger W. Dijkstra
Five Streams of Influence

1. Navigation and Logistics

1956 → present

Dijkstra's algorithm is the computational engine beneath every GPS navigation system. Google Maps, Apple Maps, Waze — all use variants of Dijkstra's algorithm (or its improvement, A*, which adds a heuristic to Dijkstra's core). The algorithm computes shortest paths through road networks containing millions of nodes, in under a second, on hardware that would have been incomprehensible to Dijkstra in 1956.

Global shipping and logistics optimization — the systems that route FedEx packages, Amazon deliveries, airline networks — use Dijkstra-class algorithms to minimize distance and cost across vast networks of locations. It is not an exaggeration to say that modern global commerce depends on an insight Dijkstra had at a café table while his fiancée waited.

2. The Internet Itself

1956 → present

The Open Shortest Path First (OSPF) protocol — one of the two main interior gateway routing protocols of the internet — runs Dijkstra's algorithm on every router that implements it. When you send a packet across the internet, the routers it passes through are continuously computing shortest paths through the network graph, updating their routing tables as links come up and go down. This is Dijkstra's algorithm running in real time, at every point of the internet's backbone.

The modern internet would not function in its current form without Dijkstra's 1956 insight. This is not hyperbole. Without efficient shortest-path computation, internet routing would require either flooding (sending every packet everywhere) or static routing tables (which could not adapt to failures). Dynamic shortest-path routing is what makes the internet resilient. Dijkstra invented the mathematical core of internet routing six years before the ARPANET existed.

3. Structured Programming — Every Modern Language

1968 → present

Python has no GOTO. Java has no GOTO. C++ has no GOTO (or deprecated goto that no one uses in serious code). Rust has no GOTO. Swift has no GOTO. Every programming language designed after approximately 1975 was designed around the structured programming principles that Dijkstra fought for. The GOTO statement is gone from serious programming. The battle is over, and Dijkstra won.

The consequences extend deeper than syntax. Every modern program is organized as functions (procedures), conditionals (selection), and loops (iteration). Every modern programmer thinks in these structures without realizing there was ever an alternative. The cognitive architecture of programming — the way programmers think about their code — was reshaped by Dijkstra's 1968 letter, which was itself building on his deeper program-correctness framework.

Modern code review practices — asking whether code is readable, understandable, testable — descend directly from the structured programming argument. The question "can you reason about what this code does?" is Dijkstra's question. It is now everybody's question.

4. Concurrent Programming — Semaphores and Beyond

1965 → present

Every operating system uses the concept of the semaphore, or a higher-level abstraction built on it. Every mutex (mutual exclusion lock) in every concurrent program is a semaphore with a maximum count of one. Every thread synchronization primitive in Java, Python, Rust, Go, C++ descends from Dijkstra's 1965 invention. The Dining Philosophers problem is still the standard pedagogical example for teaching concurrent programming — it appears in every operating systems textbook, in every algorithms course, in every treatment of deadlock and livelock.

The architects of every major operating system — UNIX, Linux, Windows NT, macOS — built their synchronization primitives on the conceptual foundation Dijkstra established. The multi-core processing revolution of the 2000s — the sudden need to write software that correctly ran on eight, sixteen, sixty-four simultaneous processors — returned Dijkstra's concurrency work to urgent relevance. His problems are now everyone's problems.

5. Formal Verification and Program Correctness

1975 → present

The field of formal verification — proving mathematically that a program does what its specification says it does — descends directly from Dijkstra's predicate transformer calculus and from Hoare Logic (itself strongly influenced by Dijkstra's work). The Coq proof assistant, the Lean theorem prover, the TLA+ specification language (developed by Leslie Lamport, a close colleague) — all sit in a direct intellectual lineage from Dijkstra's insistence that correctness is provable, not merely testable.

As software has grown to control aircraft, medical devices, nuclear facilities, financial systems, and autonomous vehicles, the ability to prove programs correct — rather than merely test them — has moved from academic exercise to practical necessity. The companies writing flight control software, the research groups developing formally verified operating systems (seL4, for instance), the engineers at AWS who use TLA+ to verify distributed system designs — all are doing what Dijkstra argued for, with the tools his tradition produced.

The Algorithm in Your Pocket

Your smartphone contains Dijkstra's algorithm in multiple places. The GPS application uses it to find routes. The Wi-Fi networking stack uses routing protocols that descend from it. The operating system kernel uses synchronization primitives (semaphores, mutexes) that Dijkstra invented. The applications running on it are written in structured programming languages whose design he shaped. And the data packets traveling to and from your phone pass through internet routers running OSPF — Dijkstra's algorithm — to find their path.

Dijkstra died in 2002, before smartphones existed. He used no email. He had no webpage. He would have regarded social media with the same precision-focused horror he brought to BASIC and COBOL. But the smartphone in your pocket is, in multiple deep senses, a product of his thinking. He did not foresee any of the specific applications. He was not trying to build an internet or enable GPS. He was asking: what does it mean for a program to be correct? How do we find shortest paths? How do concurrent processes synchronize without collision? The applications followed from the answers.

The Dining Philosophers — Still Canonical

Dijkstra invented the Dining Philosophers problem in 1965 as an illustration of the problems of concurrent resource allocation. Five philosophers. Five forks. The problem is still, sixty years later, the canonical introduction to deadlock and livelock in every operating systems course. It has been generalized, extended, and analyzed thousands of times in the academic literature. It has been used to test concurrent programming languages, verify synchronization protocols, and teach the subtleties of system design.

The reason it endures is its combination of simplicity and depth. The setup can be explained in two sentences. The solution — avoiding deadlock while preventing starvation — requires genuine insight. It is, in Dijkstra's sense, a beautiful problem: one that reveals the nature of concurrency through the most parsimonious possible setup.

The EWDs — Still Being Read

The EWD archive at the University of Texas at Austin — 1,318 documents, spanning 1958 to 2002 — is freely available online. It is actively read. Computer science students discover it and pass it around. Practitioners find in the EWDs a precision and directness that contemporary technical writing has largely lost. EWD 498, with its roster of observations about BASIC, COBOL, PL/I, and computing education, is regularly circulated and debated. EWD 1243, on notation and thought, is cited in programming language research.

His Turing Award lecture — EWD 215, "The Humble Programmer" — is still required reading in many computer science programs. Its central argument — that programming is not primarily about telling computers what to do, but about managing complexity in a way that human minds can follow — remains as relevant as it was in 1972, and arguably more urgent, now that programs have grown to scales that would have seemed impossible then.

"The humble programmer has been taught to distrust his own programming and to build his verification argument hand in hand with the program itself." — Edsger W. Dijkstra, EWD 215, "The Humble Programmer," 1972 Turing Award Lecture
Honors and the Prize in His Name

Dijkstra received the ACM Turing Award in 1972 — the highest honor in computer science — for "fundamental contributions to programming as a high, intellectual challenge." He received numerous honorary degrees from universities around the world. He was a member of the Royal Dutch Academy of Arts and Sciences and a foreign honorary member of the American Academy of Arts and Sciences.

After his death in 2002, the ACM Symposium on Principles of Distributed Computing (PODC) established the Dijkstra Prize in Distributed Computing — awarded annually for outstanding papers on the principles of distributed computing. The first award was given posthumously to Dijkstra himself, for his paper on self-stabilizing systems. The prize continues, recognizing work in the field he helped found.

What He Got Right — and What He Missed

Dijkstra got the core things right with an accuracy that has only become clearer with time. Programs should be derived from specifications, not patched until they pass tests — and the movement toward formal methods and proof-carrying code has vindicated this. Structured programming is the only serious way to write code — and the GOTO is gone. Concurrency requires formal synchronization primitives — and this is now a multi-billion-dollar problem domain. The choice of programming language shapes what you can think — and the programming language research community, thirty years later, has proved this with data.

He was wrong about some things. He was too pessimistic about the ability of heuristic and empirical methods to achieve practical results — the rise of machine learning, which he would have viewed with deep suspicion, has produced systems that work remarkably well without anything like a correctness proof. He underestimated the value of pragmatic, incremental engineering — the messy, iterative, test-driven process that produced the internet, Linux, and most of the software infrastructure of modern civilization. He thought computing should be more like mathematics. It turned out that computing is, in large part, its own thing — neither physics nor mathematics nor engineering, but something new, which perhaps requires its own standards.

But even here, his ghost is present in the debate. Every time a system fails because of an unchecked buffer overflow, because of a race condition in concurrent code, because of an untested edge case — Dijkstra is right. The failures he predicted are the failures that keep happening. The gap between what he argued for and what the industry produces is still, sixty years on, the most important problem in software engineering.

The Last EWD

EWD 1318 was written in April 2002. Dijkstra was ill; he would die in August. The document is about teaching — about what a computer science education should be, and about the beauty he saw in the discipline he had spent fifty years trying to define and defend. He did not know it was his last memo. He wrote it as he wrote all of them: carefully, by hand, in fountain pen, on lined paper.

He died on 6 August 2002, in Nuenen, the Dutch village where van Gogh had lived and painted the potato eaters a century before. His wife was with him. The bicycle was in the garage. The fountain pen was on the desk. The archive of 1,318 memos was, already, being digitized in Austin.
