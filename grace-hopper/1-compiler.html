The Premise

The Assumption Everyone Made — and She Didn't

In 1949, when Grace Hopper joined the Eckert-Mauchly Computer Corporation to work on UNIVAC, the received wisdom of the computing world was immovable: computers did arithmetic. They added, subtracted, multiplied, and divided. They operated on numbers. They did not, and could not, operate on language. If you wanted a computer to do something, you wrote in its native instruction set — binary machine code, or at best, symbolic assembly language. You, the programmer, were the translator.

Hopper thought this was absurd. Not as a matter of aesthetics, but as a matter of engineering. Programmers spent enormous amounts of time writing the same subroutines over and over — square roots, trigonometric functions, logarithms. Every program that needed a square root got a hand-written square root. She thought: why not write those subroutines once, store them in a library, and have the machine retrieve and link them on demand? The machine would be doing the translation. The programmer would be calling functions instead of writing them from scratch.

Her colleagues told her this was impossible. You couldn't have a machine write its own programs. The computer would never know what code was supposed to be data and what was supposed to be executed. The whole idea was confused, they said. Hopper disagreed. She ran a demonstration. They said it worked by accident. She ran it again.

"I had a running compiler and nobody would touch it. They told me computers could only do arithmetic."
Grace Hopper, recalling the reception of A-0 in 1952
01
1952

A-0 — The First Compiler

The A-0 system was not a modern compiler in the full sense — it was more accurately a linker and loader that assembled programs from pre-written subroutines called by a shorthand code. You wrote a three-digit number representing a mathematical function and a set of arguments; the A-0 retrieved the subroutine from its library and linked it into the executable program. But the essential conceptual leap was there: a program that writes a program. Symbolic instructions that the computer itself translated into machine code. The boundary between "code" and "data" was shown to be a design choice, not a physical law.

Hopper followed A-0 with A-1 and A-2, each more capable than the last. By A-2, she had what she would later call a true programming system: a machine that could read mathematical notation and produce executable machine instructions. Nobody celebrated. The computing establishment was skeptical. She kept building.

02
1955

FLOW-MATIC — The First English-Language Programming Language

If the A-0 proved that machines could do symbolic translation, FLOW-MATIC asked a more audacious question: why should the symbols be mathematical? Why not English? Hopper's insight was that the largest emerging market for computers was not physics laboratories, but businesses — and businesspeople did not speak machine code. They spoke English. If you wanted computers to run payroll, manage inventory, and process insurance claims, the people running those businesses had to be able to understand and eventually write the programs.

FLOW-MATIC, developed between 1955 and 1958, was the first programming language to use English words as commands. COMPARE, READ, WRITE, STOP — the vocabulary was drawn from business operations. A FLOW-MATIC program looked like a set of instructions you might give to a capable clerk. This was the whole point. Hopper was not simplifying computing; she was democratizing it. FLOW-MATIC was used in real Remington Rand UNIVAC applications and demonstrated that English-language programming was not just possible but practical.

FLOW-MATIC — c. 1957 — First English-language program (reconstructed)
(0) READ-ITEM INVENTORY-RECORD; WRITE-ITEM NEW-RECORD ;
(1) COMPARE PRODUCT-PRICE (A) WITH MAX-PRICE (B) ;
    IF GREATER GO TO OPERATION 5 ;
    IF EQUAL   GO TO OPERATION 3 ;
    IF LESS    GO TO OPERATION 2 ;
(2) TRANSFER PRODUCT-PRICE TO ADJUSTED-PRICE ;
    GO TO OPERATION 4 ;
(3) MOVE PRODUCT-NAME TO OUTPUT-BUFFER ;
(4) WRITE-ITEM REPORT-LINE ;
(5) STOP ;
03
1959

COBOL — The Language That Still Runs the World

In 1959, the United States Department of Defense convened the Conference on Data Systems Languages — CODASYL — to create a common business-oriented programming language that could run on multiple manufacturers' computers. The DoD was alarmed that every vendor had its own incompatible programming system; switching computers meant rewriting every program. Hopper chaired the Short Range Committee, which drew heavily on FLOW-MATIC. The result was COBOL: Common Business-Oriented Language.

COBOL was remarkable for its time in ways that still matter: it separated data definitions from procedures, it used a hierarchical data structure that matched how businesses actually organized their records, and it was explicitly designed to be readable by non-programmers — managers, auditors, regulators. A COBOL program was supposed to be its own documentation. You could read it and understand what the business process was doing.

COBOL shipped in 1960. By the mid-1960s it was the most widely used programming language in the world. It has never stopped being widely used. Today, an estimated 220 billion lines of COBOL remain in production — more than any other language — running approximately $3 trillion in daily financial transactions. Every time you use a bank, pay taxes, or receive a government benefit, you are almost certainly touching a system with COBOL in its core.

September 9, 1947

The Bug — First Actual Case

The word "bug" to mean a technical malfunction predates Hopper — Thomas Edison used it in 1878, and electrical engineers had used it casually for decades. But the logbook entry from the Mark II, made by Hopper's team on September 9, 1947, is the moment the term became permanently embedded in programming culture.

The Mark II was malfunctioning. The operators traced the fault to Relay 70 in Panel F — one of the thousands of electromechanical switches that made up the machine's logic circuits. They opened the panel. A moth was caught in the relay contacts, preventing them from closing properly. Someone — the exact hand is disputed, though Grace Hopper is most commonly credited — removed the moth with tweezers, taped it into the logbook, and wrote the entry that has been reproduced in textbooks ever since.

Harvard Mark II Logbook — Bureau of Ordnance Computation Project — Cambridge, Massachusetts
DATE: 0900 · 09 SEPT 1947
OPERATOR: HOPPER TEAM · PANEL F

09:43 Relay 70 · Panel F · fault on output register test
09:47 Opened Panel F · inspected relay contacts
09:51 Found moth trapped in relay contacts · removed with tweezers
09:52 Taped moth into log · [see below]
09:54 System test resumed · register output correct

"FIRST ACTUAL CASE OF BUG BEING FOUND."

— Logbook now in the permanent collection of the National Museum of American History, Smithsonian Institution, Washington D.C.

The moth and its logbook page are now at the Smithsonian's National Museum of American History. They are among the most viewed artifacts in the history of computing. The physical object — a small moth, taped to yellowing paper, in a 78-year-old notebook — is, in a certain light, the most consequential insect in human history.

Hopper did not invent the word "bug" and she never claimed to. What she did was ensure the incident was documented precisely, humorously, and permanently — in a way that reflected her deepest instinct about engineering: keep a record. Show your work. Note the anomaly. The logbook was her instrument for making the invisible visible, and she applied it to a moth with the same rigor she applied to everything else.

The Chain of Consequence

What the Compiler Made Possible

Before 1952: to use a computer, you had to speak the computer's language. That meant binary instruction codes, memory addresses in decimal, hand-linking subroutines. Programming was the exclusive domain of mathematicians and engineers with the patience to operate at the machine's level of abstraction.

After 1952: the machine could learn to speak a human-readable language. This was not a convenience feature. It was a civilizational inflection point. Every high-level programming language — FORTRAN, LISP, ALGOL, C, BASIC, Pascal, Ada, C++, Java, Python, JavaScript, Rust — exists because Grace Hopper demonstrated that a compiler was possible. Every interpreter, every transpiler, every just-in-time compiler, every domain-specific language running inside any application anywhere in the world is a descendant of A-0.

The compiler is not one of many inventions in computer science. It is the invention that made all other programming inventions accessible to non-experts — and therefore the invention that made the software industry possible.

The Four Contributions

Her Technical Legacy in Brief

1952 — A-0

The First Compiler

Proved that a computer could perform symbolic translation — reading shorthand codes and assembling a program from pre-written subroutines. The foundational proof of concept for all compiler theory.

1955–1958 — FLOW-MATIC

The First English-Language Language

Demonstrated that programming commands could be English words rather than symbols or machine opcodes. Changed the target audience for computing from mathematicians to business operators.

1959–1960 — COBOL

The Language of Global Commerce

Co-designed the language that still processes trillions of dollars in daily transactions. Readable, hierarchical, portable across vendors. 220 billion lines still in production.

September 9, 1947

The Bug

The moth in Relay 70, Panel F — logged with precision and humor — gave programmers everywhere a permanent shorthand for hardware/software failure and established the discipline of meticulous fault documentation.
